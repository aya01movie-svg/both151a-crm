# 03_AI_Guidelines — AIへのコーディング指示書

このドキュメントは、今後Claudeがこのプロジェクト（MATTY）のコードを
書く・修正する際に必ず守るべきルールをまとめたものです。

---

## 1. 基本姿勢

1. **完全なコードを出力する**：修正時は該当ファイルの完全な内容
   （または明確に特定できる完全な関数・ブロック単位）を出力すること。
   「ここを変えてください」のような断片的な指示だけで終わらせない。
   ユーザーはGitHub上のファイルを直接置き換える運用のため、差分の
   一部だけを渡すと反映漏れ・構文崩れの原因になる。
2. **推測で機能を変えない**：仕様が曖昧な場合は、複数の解釈をレポートの
   「未解決事項」に明記し、最も妥当と考えられる実装を選びつつユーザーに
   確認を仰ぐ。仕様書（過去のレポート・ユーザーの指示文）を正として、
   古いチャット内容や自分の記憶を優先しない。
3. **DBスキーマは勝手に変更・削除しない**：
   - 新しいテーブル・カラムの追加、既存カラムの削除・型変更は、
     明示的にユーザーが承認しない限り行わない。
   - 過去データの不整合（例：絵文字の自動付与バグ等）は、まず
     「表示側の正規化」で解決できないか検討する（02_Database_Schema.md
     2.3節のパターンを踏襲）。DBの値そのものを書き換えるSQLが必要な
     場合は、**実行せずSQL文だけを提示し、実行はユーザー自身に委ねる**。
   - RLSポリシー・トリガーの変更は特に慎重に。意図せず権限が広がる/
     狭まる変更をしないよう、変更前後の挙動を明記する。
4. **提出時は必ず以下を含める**（ユーザーの定型フォーマット）：
   - 変更点一覧
   - 修正ファイル一覧／追加ファイル一覧／削除ファイル一覧
   - DB変更有無
   - SQL追加有無（および既存データ修正SQLが必要か）
   - 未解決事項
   - 動作確認内容（静的チェックの内容と、実機で確認してほしい項目を
     明確に分ける）

## 2. Next.js / React の書き方

### 2.1 App Router構成
- ログイン後の全ページは `src/app/(app)/` 配下に置く
  （共通レイアウト `layout.tsx` がヘッダー・ナビを1回だけ描画するため）。
  新しいページを追加する場合も必ずこのグループ内に置くこと。
- 各ページで `getCurrentProfile()` を独自に呼び出す必要はない
  （共通レイアウトで取得済み）。ページ固有の理由でどうしても
  `profile`情報が必要な場合のみ、`getCurrentProfile()`
  （`src/lib/auth/current-profile.ts`。`React.cache()`でラップ済みなので
  同一リクエスト内での再呼び出しはコストが低い）を呼んでよいが、
  乱用しないこと。
- ページタイトル（ヘッダーの小さい見出し）は `AppShell`/`Header`に
  propsで渡すのではなく、`src/components/layout/PageTitle.tsx`の
  パス→タイトル対応表に追記する。

### 2.2 Server Actions
- ファイル冒頭に `"use server"` を書くファイルは、**async関数以外を
  exportしてはいけない**（Next.js 16のルール）。型定義・定数
  （初期state等）は隣接する `xxx-state.ts` に分離すること
  （`src/lib/actions/visits.ts` と `src/lib/actions/visits-state.ts`が実例）。
- ミューテーション（作成・更新・削除）を行うServer Actionは、処理後に
  影響のある全パスへ`revalidatePath()`を呼ぶこと（HOME・カレンダー・
  該当詳細ページなど、表示が変わりうる箇所を漏れなく洗い出す）。

### 2.3 データ取得
- 互いに依存しないデータ取得は必ず`Promise.all()`で並行実行する
  （直列awaitは画面遷移速度の劣化に直結する。v1.2で複数ページに
  この問題があり修正済み。新しいページ・機能を書く際も同じ観点で
  レビューすること）。
- 一覧系のクエリに安易な`limit()`を付けない。ユーザーから明示的に
  「上限を設けてよい」と言われない限り、全件取得を基本とする
  （イベント一覧が好例。件数が増える場合はUI側で折りたたみ等の
  工夫をする。02_Database_Schema.mdの`store_events`参照）。

### 2.4 日付・タイムゾーン処理
- サーバーの実行タイムゾーンに依存しないよう、日付文字列の生成・
  比較には必ず `src/lib/date.ts` のヘルパー
  （`toJstDateString`, `toJstTimeString`, `jstLocalToUtcIso`等）を使う。
- 曜日を計算する際、`new Date(dateString)`（日付のみの文字列）で生成した
  Dateオブジェクトに対しては **`getDay()`ではなく`getUTCDay()`を使う**
  （`dateString`はUTC正午/深夜として解釈されるため、`getDay()`だと
  サーバーのタイムゾーン設定次第で曜日が1日ズレるバグが過去に実際に
  発生した）。一方、`new Date(year, month-1, day)`のようにY/M/Dを
  個別の引数で渡すローカルコンストラクタの場合は`getDay()`で問題ない
  （ローカル値同士の比較になるため）。この使い分けを必ず守ること。

### 2.5 UI・スタイリング
- TailwindCSS v4を使用。デザイントークンは`src/app/globals.css`の
  CSS変数（`--color-navy`, `--color-gold`, `--color-danger`,
  `--color-success`, `--color-info`, `--color-warn`等）を使う。
  直接のカラーコード指定は避ける。
- ボタンの最低高さ56px、角丸12px、文字サイズ16px以上を基本とする
  （タブレット運用前提のタップしやすさ・可読性優先）。
- 视認性の低い配色（例：白背景に薄い黄色文字）は避ける。ブランドカラー
  （gold = 明るい黄色）を文字色として単体で使う場合は、背景とのコントラスト
  を必ず確認し、必要なら「濃色文字＋goldの背景バッジ」のような形に
  変更する（実際に「初回来店」表示の視認性改善で行った対応）。
- 金額・日付など重要な数値は大きく・太字（`font-black`）で強調する
  傾向がある（ユーザーからの繰り返しの要望）。

### 2.6 コンポーネント設計
- 同じ表示ロジック（例：イベント一覧の1行）が複数箇所に出てくる場合は、
  早めに小さいプレゼンテーショナルコンポーネントへ切り出す
  （`EventListItem`が実例）。フォルダ分け・折りたたみ等のUI変更を
  あとから入れやすくなる。
- 折りたたみUIは、状態管理が不要ならネイティブの`<details>`/`<summary>`
  を優先する（軽量・アクセシブル）。
- 長い一覧・フォームがある画面で「編集」等のボタンを押した結果が
  画面外に開く設計は避ける。開いた場所へ`scrollIntoView()`する等、
  反応が視覚的に分かるようにすること（過去に「ボタンが反応しない」と
  誤解された実例あり）。

## 3. 複雑な多対多更新の扱い

来店・予約の同伴者（`visit_members`/`reservation_members`）、タグ
（`customer_tags`）など、複数テーブルにまたがる登録・更新は、
既存のRPC（`create_visit_with_details`等）を通す設計になっている。

**新しい修正・編集機能を作る際、多対多の関係を含む場合は：**
1. 既存のRPCを流用・拡張できないか検討する。
2. 難しい場合は、まず「単一テーブルのカラムのみ」を対象にした
   軽量な修正機能をスコープとし、多対多部分は次回対応として
   明示的に「未解決事項」に残す（来店編集機能でこの判断を実際に行った）。
3. 安易に複数テーブルへ直接UPDATEを積み重ねる実装は避ける
   （アトミック性が失われ、中途半端な状態でエラーになるリスクがある）。

## 4. パフォーマンスに関する既知の原則

v1.2で発見・修正した「画面遷移が遅い」問題の教訓：

1. **共通レイアウトを使い、ヘッダー・ナビ等の不変部分を毎回再構築しない。**
2. **認証確認・プロフィール取得は可能な限り1箇所（共通レイアウト）で
   完結させる。** ページ個別で再度Supabase認証を呼ぶと、
   ミドルウェアでの確認と合わせて二重の通信コストになる。
3. **ローディング表示は画面全体を覆う重いオーバーレイにしない。**
   コンテンツ領域だけを軽量なスケルトン/スピナーに差し替える
   （`loading.tsx`をルートグループ内に配置し、共通レイアウトの外側を
   維持したまま中身だけ差し替わるようにする）。
4. **`force-dynamic` / `revalidate = 0` を安易に外さない。**
   過去にキャッシュに起因する「保存したのに画面に反映されない」
   バグがあり、その対策として導入された経緯がある。パフォーマンス
   改善のためにキャッシュ戦略を見直す場合は、`revalidatePath()`の
   呼び出し漏れがないか必ず全アクションを洗い出してから変更すること。

## 5. 開発コミュニケーションの前提

- ユーザーは非エンジニアに近い運用者（GitHub Web UIでファイルを
  アップロード・削除する形で反映している）。コード変更を伝える際は、
  「どのファイルをどこに置くか」を具体的なパスで明記すること。
- 複数ファイルにまたがる変更をまとめて渡す場合、**ファイルの新規追加/
  既存上書きの区別を明確にする**。特にフォルダ構成そのものを変える
  変更（v1.2のルートグループ導入等）は、「削除すべき古いフォルダ」を
  明示しないとビルドエラー（同一URLへの重複ルート）の原因になる
  （実際に発生した事例あり）。
- 複数回に分けて修正を渡す場合、ファイル間の依存関係
  （新しいフォルダ構成を前提にしたファイルなど）がある場合は、
  「どの順番で反映すべきか」を明記する。可能であれば、依存関係が
  複雑になってきた時点で、それまでの変更を1つの自己完結した
  パッケージにまとめ直して渡す（実際にこの方法で解決した事例あり）。
