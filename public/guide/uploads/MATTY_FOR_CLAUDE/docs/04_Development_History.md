## 3. 既知のハマりどころ（今後同じ轍を踏まないための記録）

1. **ルートグループ導入時のフォルダ重複によるビルドエラー**：
   `src/app/dashboard/`等の旧フォルダを削除せずに`src/app/(app)/dashboard/`
   を追加すると、Next.jsが同一URLに対応する2つのページを検出して
   ビルドエラーになる。フォルダ構成を変える変更を渡す際は、削除すべき
   旧フォルダを明示すること。
2. **複数回に分けて渡したzipの手動マージによる部品ファイル欠落**：
   ページ側の変更だけが先に反映され、それが依存するコンポーネント
   （例：`VisitEditForm.tsx`）が未反映のままデプロイされ、
   「Module not found」エラーになった。**複数ファイルにまたがる新機能は、
   できる限り1つの自己完結したパッケージにまとめて渡す**ことで解決した。
3. **Server Actionsファイルへの型・定数の混入**：
   `"use server"`ファイルに`UpdateVisitState`型と初期値定数を直接置いたら
   Next.js 16のビルドルール違反になった。既存の`xxx-state.ts`分離パターンを
   踏襲することで解決（03_AI_Guidelines.md参照）。
4. **曜日計算のタイムゾーン依存バグ**：日付文字列から生成した
   Dateオブジェクトに対し`getDay()`（ローカルタイムゾーン依存）を使うと、
   サーバーの実行環境によって曜日が1日ズレることがある。
   `getUTCDay()`への統一で解決。
5. **表示側の正規化 vs DB修正の判断**：絵文字自動付与バグのような
   「過去データに残った不整合」は、影響範囲・リスクを鑑みてまず
   表示側（データ取得関数）での正規化を優先し、DBへの書き込みを
   伴う修正は行わない、という判断を一貫して採用した。

## 4. 既知のバグ・未解決事項（現時点）

1. **来店編集機能は同伴者・タグの変更に対応していない**：
   `/visits/[id]/edit`では、来店日時・金額・チップ・支払方法・席タイプ・
   領収書・メモのみ修正可能。同伴者やタグを直したい場合は、現状
   「無効化 → 再登録」で対応する必要がある。対応するには、既存の
   `create_visit_with_details` RPCの拡張、または多対多更新を安全に扱う
   専用の仕組みが必要（02_Database_Schema.md 2.4節、
   03_AI_Guidelines.md 3節参照）。
2. **未使用コンポーネントの残存**：
   - `src/components/dashboard/MonthlyLineChart.tsx`：どこからも
     importされていない（`DashboardSearch.tsx`削除時に発見、
     明示的な削除指示がなかったため残置）。
   - `src/components/customers/HideCustomerButton.tsx`：顧客詳細ページから
     使用箇所を削除した結果、参照されなくなった（機能自体を削除する
     指示ではなかったため、コンポーネント自体は残置）。
   実害はないが、次回の整理タイミングでまとめて削除するのが望ましい。
3. **設定ページ：スタッフ休み登録ボタンのツールチップ**：
   PC・マウス操作時のみ、ボタンにカーソルを乗せるとブラウザ標準の
   ツールチップで「スタッフA」等が一瞬表示される（`title`属性）。
   タップ操作（スマホ・タブレット）では表示されない。完全に消したい
   場合は次回対応可能。
4. **画面遷移パフォーマンスの限界**：
   共通レイアウト化・認証確認の重複排除・軽量ローディングへの変更で
   体感速度は改善したはずだが、Supabaseへの実際のデータ取得時間
   （ネットワーク往復時間）自体は変えていない。`force-dynamic`を
   維持しているため、キャッシュを使った更なる高速化の余地はある
   （ただし過去に「反映されない」バグの原因になった経緯があるため、
   `revalidatePath`呼び出し漏れの洗い出しとセットで慎重に検討する必要がある）。
5. **オフライン対応の現状**：
   `OfflineBanner`・`SyncManager`・`CustomerCacheSyncManager`・
   `src/lib/offline/`（IndexedDBキュー・オンライン状態フック等）といった
   基盤コードは既に存在している（v1.1時点の「第38章」対応と思われる）。
   ただし本セッションではこれらの機能を深く検証・修正の対象にはして
   いないため、**実際にオフライン→オンライン復帰時の同期が現状どこまで
   正しく動くかは未検証**。次のTODOとして動作確認・必要なら改修を行うのが
   よい。
